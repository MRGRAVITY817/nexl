# WASM Component Composition in Nexl

This document describes how to compose multiple Nexl (and third-party) WASM components
using the [WASM Component Model](https://component-model.bytecodealliance.org/) and
`wasm-tools compose`.

---

## Overview

The WASM Component Model allows you to combine components that speak a common
interface language (WIT) into a single composed component. Nexl supports generating
WIT interface descriptions (via `nexl-wasm::wit_export`) and the `ComposeGraph` API
(via `nexl-wasm::composition`) to drive `wasm-tools compose`.

```text
┌─────────────────────────────────────────────────┐
│              nexl-component-a.wasm              │
│  exports: interface math {                      │
│    add: func(a: s64, b: s64) -> s64;            │
│  }                                              │
└───────────────────────┬─────────────────────────┘
                        │ satisfies
┌───────────────────────▼─────────────────────────┐
│              nexl-component-b.wasm              │
│  imports: interface math { … }                  │
│  exports: interface app  { run: func() }        │
└───────────────────────┬─────────────────────────┘
                        │ wasm-tools compose
                        ▼
              composed-app.wasm  (self-contained)
```

## Prerequisites

Install the `wasm-tools` CLI:

```sh
cargo install wasm-tools
```

Verify installation:

```sh
wasm-tools --version
# wasm-tools 1.x.x
```

## Workflow

### 1 — Write a Nexl component that exports an interface

Create `math-lib.nx`:

```nexl
(module math-lib
  :exports [math]
  (defn add [a : Int b : Int] : Int
    (+ a b)))
```

Compile to a WASM core module:

```sh
nexl build math-lib.nx -o math-lib.wasm
```

> **Note:** Full Component Model output requires the `--component` flag (available in M24+).
> In M23, `nexl build` produces a WASM core module. The steps below show the intended
> workflow for when Component Model support is complete.

### 2 — Write a Nexl component that imports that interface

Create `app.nx`:

```nexl
(module app
  :performs [math]        ; declares intent to use the math effect / interface
  :imports [(math add)]
  (defn main []
    (print (math/add 40 2))))
```

Compile:

```sh
nexl build app.nx -o app.wasm
```

### 3 — Compose the two components

```sh
wasm-tools compose app.wasm -d math-lib.wasm -o composed-app.wasm
```

Or using the Nexl Rust API (`nexl-wasm::composition`):

```rust
use nexl_wasm::composition::{ComposeGraph, ComponentInput};

let mut graph = ComposeGraph::new("composed-app.wasm");
graph.add_input(ComponentInput::file("app.wasm"));      // root / host
graph.add_input(ComponentInput::file("math-lib.wasm")); // dependency

let cmd = graph.compose_command().unwrap();
println!("{}", cmd.display());
// → wasm-tools compose app.wasm -d math-lib.wasm -o composed-app.wasm

cmd.execute().unwrap();
```

### 4 — Run the composed component

```sh
nexl run --wasm composed-app.wasm
# 42
```

## Importing a third-party Rust component

Any Rust crate compiled to a WASM component can be linked in. For example, a regex
engine:

```sh
# Compile the regex crate to a WASM component (using cargo-component)
cargo component build -p my-regex-lib --target wasm32-wasi

# Compose with your Nexl app
wasm-tools compose my-app.wasm -d target/wasm32-wasi/debug/my-regex-lib.wasm -o out.wasm
```

The Nexl module must declare the imported interface in its WIT bindings (generated by
`nexl-wasm::wit_import::parse_wit_interface`) or via `(wit-import "regex.wit")`.

## Exporting a Nexl component for use from another language

The Nexl WIT export API (`nexl-wasm::wit_export::generate_wit_world`) generates a WIT
world describing your module's public surface:

```rust
use nexl_wasm::wit_export::{WitModuleExport, WitExportFn, generate_wit_world};
use nexl_types::Type;

let module = WitModuleExport {
    name: "my-service".to_string(),
    exports: vec![
        WitExportFn { name: "process".to_string(), params: vec![Type::Str], ret: Type::Str },
    ],
    imports: vec![],
};
let wit = generate_wit_world(&module).unwrap();
std::fs::write("my-service.wit", wit).unwrap();
```

Output `my-service.wit`:

```wit
package nexl:my-service;

world my-service {
    export process: func(p0: string) -> string;
}
```

Other languages (Rust via `cargo-component`, Go via `wasm-tools bindgen`, Python via
`componentize-py`) can then import your component using this WIT description.

## Effect ↔ WASI capability mapping

Nexl's effect system enforces WASI sandboxing. A module can only import a WASI interface
if it declares the corresponding `:performs` effect:

| Nexl effect   | Required WASI import               |
|---------------|------------------------------------|
| `Net`         | `wasi:http/outgoing-handler`       |
| `FileSystem`  | `wasi:filesystem/types`            |
| `Console`     | `wasi:cli/stdout`                  |
| `Clock`       | `wasi:clocks/wall-clock`           |
| `Random`      | `wasi:random/random`               |
| `Sockets`     | `wasi:sockets/tcp`                 |

Violating this contract is caught at the WIT world validation step
(`nexl-wasm::effect_wasi::validate_wasi_capabilities`), before any WASM is executed.

---

## Current limitations (M23)

- `nexl build --component` is not yet implemented; WASM core output only.
- `wasi:http/incoming-handler` (server mode) requires an async runtime (planned for M24).
- `wasm-tools compose` must be installed separately (not bundled with Nexl CLI yet).
- Component Model type lifting/lowering for Nexl ADTs is partial (records and variants
  are defined; runtime lifting awaits the Component Model runtime in M24+).
